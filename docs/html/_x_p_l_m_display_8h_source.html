13 /*
14  * This API provides the basic hooks to draw in X-Plane and create user
15  * interface. All X-Plane drawing is done in OpenGL. The X-Plane plug-in
16  * manager takes care of properly setting up the OpenGL context and matrices.
17  * You do not decide when in your code's execution to draw; X-Plane tells you
18  * when it is ready to have your plugin draw.
19  *
20  * X-Plane's drawing strategy is straightforward: every "frame" the screen is
21  * rendered by drawing the 3-d scene (dome, ground, objects, airplanes, etc.)
22  * and then drawing the cockpit on top of it. Alpha blending is used to
23  * overlay the cockpit over the world (and the gauges over the panel, etc.).
24  * X-Plane user interface elements (including windows like the map, the main
25  * menu, etc.) are then drawn on top of the cockpit.
26  *
27  * There are two ways you can draw: directly and in a window.
28  *
29  * Direct drawing involves drawing to the screen before or after X-Plane
30  * finishes a phase of drawing. When you draw directly, you can specify
31  * whether X-Plane is to complete this phase or not. This allows you to do
32  * three things: draw before X-Plane does (under it), draw after X-Plane does
33  * (over it), or draw instead of X-Plane.
34  *
35  * To draw directly, you register a callback and specify which phase you want
36  * to intercept. The plug-in manager will call you over and over to draw that
37  * phase.
38  *
39  * Direct drawing allows you to override scenery, panels, or anything. Note
40  * that you cannot assume that you are the only plug-in drawing at this
41  * phase.
42  *
43  * Window drawing provides slightly higher level functionality. With window
44  * drawing you create a window that takes up a portion of the screen. Window
45  * drawing is always two dimensional. Window drawing is front-to-back
46  * controlled; you can specify that you want your window to be brought on
47  * top, and other plug-ins may put their window on top of you. Window drawing
48  * also allows you to sign up for key presses and receive mouse clicks.
49  *
50  * There are three ways to get keystrokes:
51  *
52  * If you create a window, the window can take keyboard focus. It will then
53  * receive all keystrokes. If no window has focus, X-Plane receives
54  * keystrokes. Use this to implement typing in dialog boxes, etc. Only one
55  * window may have focus at a time; your window will be notified if it loses
56  * focus.
57  *
58  * If you need to associate key strokes with commands/functions in your
59  * plug-in, use a hot key. A hoy is a key-specific callback. Hotkeys are
60  * sent based on virtual key strokes, so any key may be distinctly mapped with
61  * any modifiers. Hot keys can be remapped by other plug-ins. As a plug-in,
62  * you don't have to worry about what your hot key ends up mapped to; other
63  * plug-ins may provide a UI for remapping keystrokes. So hotkeys allow a
64  * user to resolve conflicts and customize keystrokes.
65  *
66  * If you need low level access to the keystroke stream, install a key
67  * sniffer. Key sniffers can be installed above everything or right in front
68  * of the sim.
69  *
70  */

278 /*
279  * The window API provides a high-level abstraction for drawing with UI
280  * interaction.
281  *
282  * Windows may operate in one of two modes: legacy (for plugins compiled
283  * against old versions of the XPLM, as well as windows created via the
284  * deprecated XPLMCreateWindow() function, rather than XPLMCreateWindowEx()),
285  * or modern (for windows compiled against the XPLM300 or newer API, and
286  * created via XPLMCreateWindowEX()).
287  *
288  * Modern windows have access to new X-Plane 11 windowing features, like
289  * support for new positioning modes (including being "popped out" into their
290  * own first-class window in the operating system). They can also optionally
291  * be decorated in the style of X-Plane 11 windows (like the map).
292  *
293  * Modern windows operate in "boxel" units. A boxel ("box of pixels") is a
294  * unit of virtual pixels which, depending on X-Plane's scaling, may
295  * correspond to an arbitrary NxN "box" of real pixels on screen. Because
296  * X-Plane handles this scaling automatically, you can effectively treat the
297  * units as though you were simply drawing in pixels, and know that when
298  * X-Plane is running with 150% or 200% scaling, your drawing will be
299  * automatically scaled (and likewise all mouse coordinates, screen bounds,
300  * etc. will also be auto-scaled).
301  *
302  * In contrast, legacy windows draw in true screen pixels, and thus tend to
303  * look quite small when X-Plane is operating in a scaled mode.
304  *
305  * Legacy windows have their origin in the lower left of the main X-Plane
306  * window, while modern windows have their origin in the lower left of the
307  * entire global desktop space (since they are not constrained to the main
308  * window). In both cases, x increases as you move left, and y increases as
309  * you move up.
310  *
311  */
312

730  *
731  * This routine returns the bounds of the "global" X-Plane desktop, in boxels.
732  * Unlike the non-global version XPLMGetScreenSize(), this is multi-monitor
733  * aware. There are three primary consequences of multimonitor awareness.
734  *
735  * First, if the user is running X-Plane in full-screen on two or more
736  * monitors (typically configured using one full-screen window per monitor),
737  * the global desktop will be sized to include all X-Plane windows.
738  *
739  * Second, the origin of the screen coordinates is not guaranteed to be (0,

752  * space. Thus, if you have two 1080p monitors, and monitor 2 is configured to
753  * have its bottom left touch monitor 1's upper right, your global desktop
754  * area would be the rectangle from (0, 0) to (3840, 2160).
755  *

757  * windows, rather than "floating" within X-Plane) are not included in these
758  * bounds.
